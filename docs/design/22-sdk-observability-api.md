# 22. SDK Observability API and Rich Agent State

## The Objective

As outlined in the Agent Panel design docs
(`12-agent-panel-and-control-plane.md` and `17-agent-cards-and-panel.md`), the
Swarm CLI's defining feature will be the **Agent Panel**—a dynamic UI that
visualizes the concurrent activity of all active agents.

For this UI to be the "showcase of the very best software built to date in
helping users get a realtime view", the SDK's core API must support deep,
structured observability. Currently, `swarm.Chat()` yields a stream of
`ChatEvent` objects where the payload is merely a monolithic `Content string`.

This is insufficient. To render an "Agent Card" that pulses when a tool is
executing, displays the tool's runtime arguments, and turns red on a failure,
the UI must receive **structured JSON state updates**, not unstructured log
strings.

## The Proposed Observability Architecture

### 1. The `AgentState` struct

We need a unified structure to track the lifecycle of a specific agent's
execution.

```go
type AgentState string

const (
	AgentStateSpawning  AgentState = "spawning"
	AgentStateThinking  AgentState = "thinking"
	AgentStateExecuting AgentState = "executing"
	AgentStateWaiting   AgentState = "waiting" // Blocked on Human-in-the-Loop or Dependency
	AgentStateComplete  AgentState = "complete"
	AgentStateError     AgentState = "error"
)
```

### 2. Upgrading `ChatEvent` to `ObservableEvent`

The SDK stream must be upgraded to emit rich, fully-typed telemetry events.

```go
type ObservableEvent struct {
    // Identity
    Timestamp time.Time
    AgentName string
    SpanID    string

    // Core State Update
    State AgentState

    // Execution Context
    ToolName string // If executing a tool
    ToolArgs map[string]any // Structured arguments for the UI to display cleanly

    // Output
    Thought      string // The agent's internal reasoning
    FinalContent string // The final textual output
    Error        error  // Formal error objects for the UI to highlight

    // Observers
    ObserverSummary string // Semantic summary generated by the lightweight observer model
}
```

### 3. Separation of Concerns: The Observer Model

As designed in doc 12, raw tool execution strings (like
`{"recursive": true, "dir": "."}`) shouldn't just be dumped on the screen. The
SDK should intercept these raw telemetry events and use a very fast/cheap
model (like `gemini-2.5-flash-8b`) to construct an `ObserverSummary` (e.g.
"Scanning the local filesystem for target files..."). This semantic summary is
what populated the `Content` of the Agent Card.

### 4. Implementation Plan for the SDK

1. **Redefine `ChatEvent`:** We will deprecate the old primitive `ChatEvent`
   struct in `pkg/sdk/swarm.go` and replace it with a hyper-detailed
   `ObservableEvent` (or greatly expand the existing `ChatEvent` fields).
1. **Plumb the `executeSpan` Loop:** Modify the event loop inside
   `executeSpan` to populate these new fields. When `part.FunctionCall`
   triggers, it must attach `part.FunctionCall.Args` cleanly natively to the
   event so the UI can construct tool breakdowns.
1. **The `AgentPanel` Bubble Tea Grid:** We will build a new Bubble Tea model
   component in `cmd/swarm/interactive.go` that explicitly reads these
   structured events and updates its internal Go Map of active Agent Cards,
   re-rendering the terminal layout on every tick.
